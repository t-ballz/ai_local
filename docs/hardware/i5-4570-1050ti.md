# i5-4570 + GTX 1050 Ti 4 GB

> Hardware knowledge sourced from Intel ARK, NVIDIA product specs, and llama.cpp compatibility docs. Token speed estimates are rough approximations based on bandwidth calculations — benchmark your own setup to verify.

## System specs

| Component | Spec | Notes |
|-----------|------|-------|
| CPU | Intel Core i5-4570 (Haswell, 4C/4T, 2013) | AVX2, FMA3 — helps with quantized inference |
| RAM | 24 GB DDR3-1600 | See dual-channel note below |
| RAM bandwidth | ~25.6 GB/s (theoretical peak, dual-channel) | ~12.8 GB/s if single-channel |
| GPU | NVIDIA GTX 1050 Ti (Pascal, GP107) | CUDA 6.1 |
| VRAM | 4 GB GDDR5 | 128-bit bus |
| VRAM bandwidth | ~112 GB/s | |
| PCIe | PCIe 3.0 x16 | ~15 GB/s practical throughput |

!!! note "Dual-channel RAM"
    Whether your 24 GB runs in dual-channel depends on DIMM configuration. The i5-4570 (LGA1150) supports 2-channel DDR3 across 4 slots. Three identical 8 GB DIMMs in slots 1/2/3 typically run in flex mode (~partial dual-channel). Two matched pairs (e.g. 2×8 + 2×4) give full dual-channel. Check your BIOS memory info — dual-channel nearly doubles effective bandwidth.

!!! warning "Pascal GPU limitations"
    The GTX 1050 Ti is CUDA 6.1 (Pascal). llama.cpp supports it, but:

    - **No flash attention** — requires CUDA ≥ 8.0 (Volta+). KV cache operations are slower.
    - **Limited FP16 throughput** — Pascal does FP16 at the same rate as FP32 (no 2× speedup). Modern GPUs get 2× with FP16.
    - **No INT8 GEMM** — requires Turing+. Q8 quant won't accelerate on this GPU.

    For this card, Q4_K_M is the best quantization choice.

## Bandwidth constraints summary

Understanding the bottlenecks is key to making the right model choices for this rig:

| Link | Practical bandwidth | Bottleneck for |
|------|-------------------|----------------|
| VRAM (GDDR5) | ~112 GB/s | Weight reads when fully in VRAM |
| DDR3 dual-channel | ~25 GB/s | CPU inference, expert weight reads |
| PCIe 3.0 x16 | ~15 GB/s | GPU↔CPU activation transfers |

The critical insight: **PCIe 3.0 is slower than DDR3**. When the GPU fetches expert weights from RAM, they travel at ≤15 GB/s over PCIe — lower than the ~25 GB/s the CPU can read them directly. GPU offloading still helps because the attention and KV computation on the GPU is much faster than CPU, and only small activations (not weights) cross the PCIe bus per layer. But the benefit is smaller than on a modern PCIe 5.0 system.

## Hybrid: attention in VRAM, rest in RAM

For 7B+ models that don't fit in 4 GB VRAM, llama.cpp can offload the first N layers to GPU and run the rest on CPU. The strategy from the MoE offloading research applies here too: put as many layers as fit in VRAM — particularly the early layers whose KV cache benefits most from GPU bandwidth.

!!! note "Approximate speeds"
    These are rough estimates based on DDR3-1600 bandwidth and CPU core count. Actual speeds depend on DIMM config, context length, and llama.cpp version.

| Model | Total GGUF Q4 | GPU layers (est.) | CPU layers | Approx speed |
|-------|--------------|------------------|------------|-------------|
| Qwen3-8B | ~5 GB | ~20/36 layers in VRAM | ~16 in RAM | ~4–7 t/s |
| Qwen3-14B | ~9 GB | ~8/40 layers in VRAM | ~32 in RAM | ~2–4 t/s |
| Gemma 3 12B | ~7.5 GB | ~10/46 layers in VRAM | ~36 in RAM | ~2–4 t/s |

Use `--fit` in llama.cpp to let it determine the layer split automatically:

```bash
llama-server -m qwen3-8b.Q4_K_M.gguf --fit -c 2048
```

Keep context length short (`-c 2048` or lower) — KV cache takes VRAM away from model layers.

## MoE offloading analysis for this rig

The general MoE offloading strategy (attention/KV in VRAM, routed experts in RAM) still applies, but the constraints are tighter here than on modern hardware.

### Qwen3-30B-A3B — the one viable large MoE option

Total model: ~17–19 GB Q4_K_M. With 24 GB RAM and 4 GB VRAM:

- Non-expert tensors (attention, KV, router, shared experts): these are the priority for VRAM. For a 30B model they are likely ~1.5–3 GB — which fits in 4 GB VRAM alongside a short KV cache.
- Routed expert weights (~15–17 GB): stay in RAM.
- Active params per token: ~3B (the same 3B shifts every token across the 30B pool).

The llama.cpp flags:

```bash
llama-server -m qwen3-30b-a3b.Q4_K_M.gguf \
  -ngl 99 \
  --n-cpu-moe \
  -c 1024
```

`-ngl 99` sends all layers to GPU; `--n-cpu-moe` moves routed expert weights back to RAM. Keep context short — at 24 GB total RAM there is little headroom.

!!! warning "Realistic speed expectations"
    Expert weights travel over PCIe 3.0 (~15 GB/s) from RAM to GPU per token. DDR3 bandwidth (~25 GB/s) and PCIe 3.0 together mean expert reads are slow. Expect **~1–3 t/s** — usable for reading/reasoning tasks but not interactive chat.

!!! tip "When CPU-only might be faster"
    Community experience (u/Pattinathar, r/LocalLLaMA) suggests that on systems where PCIe bandwidth is near or below RAM bandwidth, a full CPU-only run (`-ngl 0`) can match or beat hybrid GPU offload for some MoE models. Benchmark both and keep the faster setting.

### Smaller MoE options

| Model | Total GGUF Q4 | Fits in 24 GB RAM? | Notes |
|-------|--------------|-------------------|-------|
| Qwen3-30B-A3B | ~17–19 GB | Marginal (tight) | Best MoE option for this rig |
| Gemma 4 26B-A4B | ~16 GB | ✓ | MoE; 4B active — see caveat |
| Llama 4 Scout (109B) | ~65 GB | ✗ | Way over RAM budget |

!!! warning "Gemma 4 26B-A4B caveat"
    Gemma 4's layers are reported to be very large (~400 MB each). On a 4 GB VRAM card, it's unlikely enough non-expert tensors fit to make hybrid offloading worthwhile. Try `-ngl 0` (CPU-only) first and compare against a small `-ngl` value.

## Practical recommendations

| Goal | Model | Mode | Expected speed |
|------|-------|------|---------------|
| Fastest daily use | Qwen3-4B Q4_K_M | Full GPU | ~10–20 t/s |
| Best GPU-resident quality | Qwen3-4B or Gemma 3 4B | Full GPU | ~10–20 t/s |
| 7B quality, slower | Qwen3-8B Q4_K_M | Hybrid (-ngl ~20) | ~4–7 t/s |
| Reasoning tasks (slow) | DeepSeek R1-Distill-Qwen-14B | CPU-only | ~1.5–3 t/s |
| Large MoE (overnight tasks) | Qwen3-30B-A3B | Hybrid (--n-cpu-moe) | ~1–3 t/s |

For daily interactive use on this hardware, **stay at 4B or smaller** for a responsive experience. Use larger models with CPU offload for batch tasks where speed is less critical.
